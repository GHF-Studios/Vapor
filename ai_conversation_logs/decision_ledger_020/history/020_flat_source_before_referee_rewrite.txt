## Already Answered Constraints

These should not be re-litigated unless you explicitly reopen them.

1. Owner answers are high-authority raw input, but not automatic doctrine until integrated.
2. Questions are the default mode for broad uncertainty.
3. File edits, code generation, docs, formatters, tests, and validation require explicit approval.
4. New active repos should avoid `/docs` unless a repo later proves it needs one.
5. Root `README.md`, crate rustdoc, useful module-local rustdoc, and clear local names are the active documentation surfaces.
6. TOML comments should be rare and local.
7. The old glossary/RFC/diagram material is quarry/archeology, not active Vapor doctrine.
8. The old glossary still carries useful signal and should not be dismissed as worthless.
9. The glossary should be minimally mutated now, not actively grown as the main work surface.
10. Vapor is the root ecosystem layer for generic Engines, Games, Mods, packs, SDK/launcher-facing protocol, capability/Rhai authoring, validation, and distribution.
11. Vapor must not depend on, name, or assume concrete Spacetime Engine or Loo Cast Game internals.
12. Vapor may name generic roles such as Engine, Game, Mod, Extension Mod, Enginepack, Gamepack, Modpack, and Packagepack.
13. Vapor-SDK owns authoring, project/template/toolchain, validation, publishing, and distribution workflows.
14. Vapor-Launcher owns install, select, validate, compose, and launch workflows for players and modpack authors.
15. Publishing belongs to SDK by default; launcher may compose packs from existing content where explicitly designed.
16. Vapor-Examples is official example/test content and should consume Vapor like an external project where practical.
17. Vapor-Examples should not depend on Vapor-SDK or Vapor-Launcher as ordinary code dependencies.
18. Development dependencies may use Git/branch dependencies; release dependencies should default to crates.io/proper distributed artifacts.
19. Branch dependencies are development-only, not release.
20. `0.5.0` is the rolling target version for now.
21. `Packagepack`, `Enginepack`, `Gamepack`, and `Modpack` are comparatively stable terms.
22. `Vapor Platform`, `Capability Profile`, `Capability Declaration`, `Rhai Asset`, and `Runtime Lock` were explicitly still suspect in LC-Q005.
23. Steam is the required external substrate for current public/published Vapor usage.
24. Local/offline authoring should be possible without live Steam auth.
25. Steam-specific terms should be mirrored where they matter, not hidden behind fake generic terms.
26. Steam/Workshop/network failures should become recoverable diagnostics where possible.
27. No Phase 3 hostile-code sandbox claim.
28. No Phase 3 content signing; Steam identity plus fingerprints were considered sufficient at the time.
29. Rustup-vs-standalone-archives is not decided. It is evidence-driven.
30. `x86_64-pc-windows-gnu` vs `x86_64-pc-windows-msvc` is not decided. It is evidence-driven.
